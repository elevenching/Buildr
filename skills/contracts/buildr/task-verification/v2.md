# 任务验证

## Purpose

根据当前 workspace 或 Project 的权威政策，以最低充分范围执行验证，将所需保证、结果、候选身份和真实 wall-clock 耗时作为可复用证据返回。

## Consumer Obligations

consumer 必须提供任务/change、operation（`inspect`、`execute` 或 `cleanup`）、发布或高风险意图、变更路径、候选 identity、Project context 与已有 evidence。consumer 不得自行选择测试或预先指定正式验证级别；无法提供稳定 identity 时必须明确限制。consumer 使用完 transient evidence 后必须通知 provider 清理。

## Minimum Guarantees

provider 必须优先解析已登记 Project 的可选 `verification.yml`，再从 Rules、Project context、OpenSpec artifacts 或项目文档解析 legacy policy。`minimal` 仅用于实现循环内的快速反馈；provider 必须根据任务上下文、Project policy、风险和用户意图决定 `requiredAssurance: affected | candidate`：普通开发与普通收尾为 `affected`，发布、高风险或用户明确要求完整验证为 `candidate`。

provider 必须按 maturity、stage、enforcement、applicability、环境、副作用、授权与显式 `supersedes` 选择最低充分能力；不得猜测命令、按技术栈推断覆盖、机械执行全部测试、把较低级别结果升级为 Candidate，或要求 Git worktree 才能工作。没有声明时必须保持 legacy discovery，不得新增失败、阻塞或环境启动。

provider 必须区分只核对 evidence 的 `inspect`、启动验证命令的 `execute` 和处理 evidence 生命周期的 `cleanup`。实现内容变化后旧 evidence 失效，但重新执行的仍是同一 `requiredAssurance`，不得机械升级为 Candidate。

provider 同时支持用户直接验证、初始化/更新测试声明和能力演进意图。

## Effects and Authorization

执行任务范围内声明为 `none` 或 `local-temporary` 且不访问外部系统的常规构建/测试属于验证动作；启动服务、修改外部系统、部署环境、持久业务数据、共享状态或 unknown effects 必须另行授权。初始化或增量补充声明只能产生 discovered 或 trial/advisory 候选，不得自动执行新环境测试或提升 stable/required。provider 只能清理自己创建、已被 consumer 使用完的精确 transient run。

## Result Evidence

返回 `operation`、`requiredAssurance`、`level`、`status`、`policySources`、`policyMode`、`candidateIdentity`、`availableCapabilities`、`selectedCapabilities`、`skippedCapabilities`、`blockedCapabilities`、`coverageSummary`、`environmentReadiness`、`authorizationDecisions`、`candidateCompleteness`、逐项 `checks`、`totalDurationMs`、`timingSource`、`slowestCheck`、`failedChecks`、`skippedChecks`、`reusable`、`taskVerificationExecuteCalls`、`candidateExecutorCalls`、可用时的 `evidenceReference`，以及 `evidenceRetention`、`cleanupAfter`、`cleanupStatus` 和可用时的 `cleanupReference`。

正式 consumer evidence 必须 `status: passed`、`level == requiredAssurance`、identity 匹配且 `reusable: true`。Candidate 还必须 `candidateCompleteness: confirmed`。`inspect` 与 `cleanup` 的两个 execute count 必须为 `0`；`timingSource` 只能是已核对 summary 的 `verifier-reported` 或进程外单调时钟的 `wrapper-measured`，不得相加并行检查耗时推算整体 wall-clock。

`evidenceRetention` 至少区分 `transient`、`caller-managed` 与 `session-only`。

## Decision Points

无法确认 operation、政策、required assurance、候选 identity 或 summary 归属，验证仍在运行、必要检查失败，或验证会触发未授权副作用时必须停止并返回 `incomplete`/`failed`。`inspect` 或 `cleanup` 不得启动 verifier；无法证明 cleanup reference 安全时必须保留现场。

## Allowed Variations

provider 可以选择项目适用的命令发现、声明扫描、执行工具、非 Git identity、诊断格式、证据存储和安全清理实现，但必须保持零配置兼容、`minimal` 内部反馈与 `affected`/`candidate` 正式保证语义、真实 wall-clock、失败状态、证据生命周期和用户披露字段兼容。`task-worktree-lifecycle` 不拥有验证政策或与 checkout 无关的 evidence 清理。
