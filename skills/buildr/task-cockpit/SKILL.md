# Task Cockpit Skill

本 Skill 由 Agent 单向维护整个任务面向用户的可视化驾驶舱。驾驶舱不是 OpenSpec change 的翻译，也不替代 specs、代码、验证或外部系统事实。

## 判断创建还是维护

满足任一条件时创建或继续维护驾驶舱：

- 用户明确要求任务可视化、驾驶舱、整体进度、任务全景或长期跟踪。
- 任务跨多个阶段、change、服务、代码仓或团队。
- 任务存在外部依赖、交叉依赖、等待事项或无法一次完成的后续阶段。
- 任务预计跨会话持续推进，或需要多次用户判断。
- 单个 `tasks.md`、当前 change 或对话无法表达完整任务生命周期。

简单、短时、无持续跟踪价值的任务继续用简洁对话汇报，不为形式完整创建空洞页面。

## 解析任务与位置

1. 从 Buildr workspace root、Project registry 和任务实际范围解析拥有任务的 Project，不根据当前目录猜测。
2. 使用稳定、简短的 kebab-case `task-id`。一个驾驶舱可以关联多个 OpenSpec change。
3. 查找 `projects/<project>/openspec/knowledge/task-cockpits/*-<task-id>.html`；存在时继续更新，不按更新时间创建新文件。
4. 首次创建时使用 Project 所在环境的本地日期，路径固定为：

   ```text
   projects/<project>/openspec/knowledge/task-cockpits/yyyy-MM-dd-<task-id>.html
   ```

5. 从本 Skill 的 `assets/task-cockpit-template.html` 复制模板到目标位置，替换内嵌 `#cockpit-data` JSON。保持单文件，不引用 CDN、远端脚本、远端字体或远端图片。

## 核实任务事实

按任务相关性读取，不做无关全量审计：

- 用户已经确认的目标、边界和判断。
- 当前及历史 OpenSpec proposal、design、specs、tasks 和 CLI status。
- 相关代码、提交、测试和验证结果。
- 外部团队、接口、审批、发布或联调依赖。
- 既有驾驶舱中的历史阶段与稳定任务身份。

权威来源冲突时先修正驾驶舱，不用驾驶舱覆盖权威事实。改变 requirement、状态流、API、权限或数据语义的决策先按 `task-triage` 维护 OpenSpec；普通进度、阶段关系、外部等待和验证结果可直接维护驾驶舱。

## 组织内容

### 首页

首页必须让普通用户一眼看懂：

- 我们在做什么。
- 当前结论和阶段。
- 已完成、正在做、下一步。
- 当前阻塞或需要用户关注什么。
- 整体方案的极简链路。

首页先写结论再写原因。每张卡片保持短小；不要以 API、类名、数据库字段、大表格或代码作为首页主体。

### 推进

按整个任务而不是单个 change 展示：

- 已完成、进行中、等待、未开始的阶段。
- 阶段内的关键任务和依赖。
- active/archive change、code-only 工作和外部协作关系。
- 可核实的“阶段 N/M”“任务 N/M”，不猜测百分比。

已完成阶段保留摘要，默认减少细节；当前阶段和阻塞项高亮。

### 方案

用普通语言展示整体思路、业务流程、职责边界、关键选择和明确非目标。优先使用简短流程或对照卡，不机械复制 `design.md`。

### 技术细节

将以下内容后置：API、状态映射、数据结构、关键模块与文件、简略代码、测试结果和证据链接。只保留帮助用户或后续 Agent 理解实现的关键细节，不堆原始命令流水或完整日志。

## 更新节点

在以下节点核实事实并更新同一文件：

- 首次创建。
- 目标、范围或整体方案变化。
- 任务阶段变化或完成一个任务组。
- 出现、变化或解除阻塞。
- 用户完成关键判断。
- 用户询问进度。
- 验证结果改变结论。
- 任务暂停、恢复或完成。

没有改变任务认知的短暂命令或检查不机械刷新。

## 页面维护约束

- 用户只通过 Agent 对话参与；checkbox、状态 chip、进度条和按钮只读，不回写任务事实。
- 只修改模板内嵌 `script#cockpit-data` 的 JSON 和确有必要的任务专属文案；保持模板结构、响应式样式和导航行为稳定。
- JSON 中不放 secret、token、cookie、个人敏感信息、完整思考过程或无关原始日志。
- `updatedAt` 使用实际更新时间；状态未核实时明确写“待核实”，不伪造进度。
- 任务完成后保留稳定入口和历史阶段，不因 change archive 移动驾驶舱。

## 检查

创建或实质更新后：

1. 确认文件名匹配 `yyyy-MM-dd-<task-id>.html`，且位于正确 Project 的 `openspec/knowledge/task-cockpits/`。
2. 检查内嵌 JSON 可解析，页面没有外部网络依赖。
3. 用本地浏览器检查桌面和窄屏布局；确认默认首页聚焦，导航可用，技术内容后置。
4. 对照权威来源检查当前状态、任务数量、阻塞、验证结论和链接。
5. 确认 HTML 只读，没有修改任务状态的事件处理或外部写回。

## 面向用户的回复

驾驶舱首次创建、实质更新、用户询问进度、任务暂停或完成时，回复提供可点击入口并包含：

```text
任务驾驶舱：[查看 <任务名称>](<absolute-path>)
位置：<workspace-relative-path>
当前状态：<一句话状态>
驾驶舱已更新：<实际时间>
```

当前 runtime 不支持本地 Markdown 链接时仍提供准确绝对路径和 workspace 相对路径。未成功更新时不得声称已更新，改为说明原因和下一步。
