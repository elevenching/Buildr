## Context

Project 与 Service 详情已经与编辑 URL 分离，但仍使用多种视觉容器和详情内的关联资源导航。这让同一资源的事实被拆散，也让列表和详情同时承担跨资源导航。现有 HTTP API、revision CAS 与 prompt-only Agent 边界必须保持不变。

## Goals / Non-Goals

**Goals:**

- 详情页以连续、统一的标签和值作为主阅读模型。
- 列表操作列成为详情、编辑与关联资源跳转的唯一导航入口。
- 保留详情页头的资源身份与独立编辑入口，不让详情页承载关联资源卡片或列表。

**Non-Goals:**

- 不改变详情 URL、写 API、metadata 字段或保存冲突处理。
- 不改变 Change 的 artifact 阅读与 Agent prompt 行为。
- 不添加全局搜索、批量操作或新的资源关系。

## Decisions

### 详情使用单一事实列表

Project 与 Service 详情的稳定事实采用同一 `dl` 行样式；技术信息继续折叠，但展开后使用相同标签和值行。这样保留技术信息的次级层级，同时消除概览卡、说明卡和关联卡之间的展示差异。

### 关联导航只出现在目录操作列

项目行提供详情、编辑、服务和变更入口；服务行提供详情、编辑和所属项目入口。详情页不再显示服务、变更或项目跳转。列表是资源集合和资源间导航的唯一位置，因此操作形式一致且可在返回目录后重用。

### 编辑入口保持在详情页头

“编辑”不是关联资源导航，而是当前资源的明确主操作；它继续指向独立编辑 URL。移除该入口会迫使用户额外返回目录才能修改，降低已建立的查看到编辑流程效率。

## Risks / Trade-offs

- [详情页缺少跨资源快捷入口] → 返回链接保持可见，且目录操作列提供同等且更一致的入口。
- [事实列表变长] → 技术信息默认折叠，主事实只保留名称、说明、代码、来源和状态。
- [目录操作列变宽] → 保持低强调文本链接并允许表格容器在窄屏独立横向滚动。
