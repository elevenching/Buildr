## Context

Buildr 当前已能管理工作区规则、项目、服务索引、OpenSpec、practices、技能和 Agent 运行环境投射。规则和技能已有明确源资产与运行环境投射边界，但团队依赖的外部命令行工具仍主要靠口头说明或个人机器状态传递。

这次变更补齐命令行工具清单能力，并明确规则、技能、命令行工具这三类 Agent 直接工作资产的边界：规则表达必须遵守的约束，技能表达任务执行方法，命令行工具清单表达当前作用域期望具备的外部命令能力。

“工具型资产”只是对这三类资产的概念性统称，用于帮助理解它们都服务于 Agent 工作过程；本变更不引入统一生命周期、统一解析命令或统一管理模块。

## Goals / Non-Goals

**Goals:**

- 明确产品原则：Buildr 管团队资产，不接管个人机器。
- 在默认工作区规则中加入简练、可执行的工具型资产维护规则。
- 新增根级命令行工具清单，用于登记团队期望使用的外部命令行工具。
- 新增命令行工具清单检查能力，并让 doctor 聚合检查结果。
- 区分工作区源资产错误和当前运行环境差异：清单非法是错误，本机缺工具或版本不满足是警示。
- 更新 Buildr 技能，让 Agent 知道如何维护规则、技能和命令行工具清单。
- 将三层作用域、覆盖语义、来源链和权限控制等扩展记录到产品 roadmap，而不纳入本次 MVP 实现。

**Non-Goals:**

- 不新增统一工具型资产管理入口，例如 `assets resolve` 或等价命令。
- 不实现 project/service 级命令行工具清单解析、叠加或覆盖。
- 不实现 `commands render`。
- 不实现命令行工具安装，也不维护平台化安装方案。
- 不保存或检查 token、cookie、登录态、个人认证状态或个人私有配置。
- 不维护技能到命令行工具的依赖图或反向索引。
- 不在本变更中实现工具型资产权限控制系统。
- 不默认引入 `COMMAND.md`。
- 不在本变更中设计 `capabilities/` 抽象。

## Decisions

### 1. 三类资产保持边界分工，不做统一资产命令

规则、技能、命令行工具都直接服务于 Agent 工作，但它们的职责不同：

- 规则回答“必须遵守什么”。
- 技能回答“遇到某类任务怎么做”。
- 命令行工具清单回答“当前作用域期望本机具备哪些外部命令能力”。

因此，本变更只在规则和技能中增加维护引导，在 CLI 中增加命令行工具清单检查，不新增 `assets resolve` 或统一资产管理命令。

### 2. 默认规则强调源资产优先

默认工作区 `AGENTS.md` 中新增的规则只负责触发和路由：识别任务是否涉及规则、技能、命令行工具；如果涉及，就先使用 Buildr 技能，再由 Buildr 技能指导源资产维护和同步动作。

建议规则文本：

```md
## Buildr 资产维护

- Buildr 管理团队资产，不接管个人机器；Agent 运行环境、本机状态和临时提示都不是资产源。
- 当任务涉及新增、修改或删除需要沉淀或复用的规则、技能、命令行工具时，必须先使用 Buildr 技能。
- 规则、技能、命令行工具的变更必须先维护到 Buildr 工作区源资产，再按需同步到 Agent 运行环境或本机。
- Buildr 技能不可用时，运行 `buildr bootstrap guide`。
```

“需要沉淀或复用”用于区分临时任务操作和应写回 Buildr 的资产变更，避免把一次性命令、临时提示或个人机器状态误写成团队资产。具体源资产位置、命令边界和同步动作由 Buildr 技能说明。技能是目录级资产，`skills/<skill-id>/` 至少包含 `SKILL.md`，也可以包含该技能需要的配套文件。这里的“当前 scope”由 Agent 根据任务所在工作区、Project 或 Service 判断。Buildr 产品仓中的 `rules/AGENTS.workspace.md` 只是随包源；用户工作区初始化后实际维护的是自己的规则、技能和命令行工具源资产。

### 3. 源资产维护命令与运行环境投射命令分离

Buildr 命令需要区分两类动作：

- 源资产维护：修改 Buildr 工作区中的规则、技能、命令行工具清单。
- 同步到运行环境或本机：把规则和技能投射到 Agent 运行环境，或检查命令行工具清单与本机差异后由 Agent 协作安装。

已有命令语义保持不变：

- `buildr skill install <agent>`：安装或修复 Buildr 产品内置技能到目标 Agent 运行环境。
- `buildr rules render <agent>`：把规则源资产投射到需要桥接文件的 Agent 运行环境。
- `buildr skills render <agent>`：把工作区或项目级技能源资产投射到支持技能运行环境的 Agent。

后续源资产维护命令使用复数模块名，避免和现有 `skill install` 混淆：

- `buildr rules add/remove`：维护规则源资产。
- `buildr skills add/remove`：维护 `skills/manifest.yml` 和 `skills/<skill-id>/` 技能目录。
- `buildr commands add/remove`：维护 `commands/manifest.yml`。
- `buildr commands check`：检查命令行工具清单与当前本机环境差异。

暂不新增 `rules check` 或 `skills check`。规则和技能是否已同步到当前 Agent 运行环境，由现有运行环境检查、规则渲染、技能渲染和 doctor 聚合提示承担；命令行工具因为最终落在本机，所以需要独立 `commands check`。命令行工具不提供 render，也不提供 install。

这些命令为 Buildr 技能提供更稳定的操作入口；本次 MVP 仍可允许 Agent 直接编辑源文件，但设计上应把“维护源资产”和“投射到运行环境或安装到本机”分开。

### 4. MVP 只实现根级命令行工具清单

命令行工具清单的后续模型应支持 root / project / service 三层作用域，但本次 MVP 只实现根级入口：

```text
commands/manifest.yml
```

project/service 级清单、同 `id` 下层整项覆盖、来源链输出和权限控制进入产品 roadmap，不作为本次验收范围。MVP 的数据结构和文案不应堵死这些扩展。

### 5. 命令行工具使用清单和检查，不使用渲染

技能是 Markdown 文件，Buildr 可以保存源文件并投射到 Agent 运行环境。命令行工具主体是外部分发程序，最终安装在个人机器，所以 Buildr 只登记清单并检查本机是否满足声明。

MVP 清单示例：

```yaml
schemaVersion: buildr.commands/v1
commands:
  - id: lark-cli
    name: 飞书 CLI
    executable: lark-cli
    purpose: 团队用于访问飞书开放能力的标准命令行工具
    version:
      constraint: ">=0.1.0"
      args: ["--version"]
    install: 官方安装文档：https://example.com/lark-cli
```

字段边界：

- `id` 表示命令行工具资产身份，用于引用、覆盖和诊断；它可以和 `executable` 相同，但不强制相同。
- `executable` 表示本机可执行命令名。
- `version` 可选；如果填写版本约束，必须填写结构化检查参数。
- `install` 是最小安装提示或官方链接，不是 Buildr 可执行安装计划。

Buildr 不提供 `commands render`，也不自动安装命令行工具。

### 6. 版本检查不执行任意 shell 字符串

版本检查采用可执行文件加参数数组的结构化模型。Buildr 只调用已声明的 `executable` 和参数，不解释清单中的任意 shell 字符串。

例如：

```yaml
executable: lark-cli
version:
  constraint: ">=0.1.0"
  args: ["--version"]
```

MVP 使用默认版本输出解析，优先提取第一个语义化版本号；不支持自定义正则。解析不了时输出警示“无法判断版本”，不把它视为源资产错误。

### 7. 检查结果区分错误和警示

`commands check` 和 doctor 中的命令行工具清单检查需要区分两类问题：

- 源资产错误：清单不可解析、字段非法、版本约束格式非法等，状态为 `error`。
- 当前运行环境差异：命令不存在、版本不满足、版本无法判断等，状态为 `warning`。

命令行工具差异不阻断 doctor 总体通过，但必须警示，并说明当前运行环境与工作区声明的差异。`--json` 输出应包含标准化状态，至少支持 `ok`、`warning`、`error`。

安装建议的事实来源必须来自命令行工具清单。默认人类可读输出可以摘要差异和下一步，`--json` 输出完整结构化结果。

### 8. 安装来源保持最小，不做平台模型

命令行工具清单可以提供最小安装提示或官方链接。Buildr 不维护 macOS、Linux、Windows、包管理器或平台矩阵，也不自动执行安装建议。

当用户希望补齐本机环境时，Agent 可基于清单提供的可信起点、当前机器状态和用户授权自行协作安装或升级。

### 9. 技能可以引用命令行工具，但不维护依赖图

技能表达如何完成某类任务，因此可以在正文中引用命令行工具清单中的工具。MVP 不新增结构化依赖字段，不维护技能到命令行工具的反向依赖索引，也不做依赖图校验。

默认使用策略是乐观执行、失败后检查：如果技能步骤执行时出现命令不存在、版本不满足或明显由工具缺失导致的失败，Agent 再运行 `commands check` 或 `doctor --json`，向用户说明当前运行环境与工作区声明的差异。

如果某个技能后续步骤成本高、会修改外部系统或版本差异可能导致错误结果，技能正文可以显式要求执行前检查。

### 10. 默认清单是否为空取决于随包内置能力

默认初始化时，技能清单和命令行工具清单是否为空，都取决于 Buildr 是否随包提供对应内置能力。

当前 Buildr 随包提供 OpenSpec 相关技能，因此默认技能清单不是空的。当前 Buildr 不随包提供任何默认外部命令行工具声明，因此默认命令行工具清单为空。

Buildr 自身不默认声明到命令行工具清单中。Buildr 的存在和版本属于 bootstrap 或 doctor 顶层状态，不属于工作区声明的外部命令行工具资产。

### 11. 后续路线项进入 roadmap

以下方向已记录到产品 roadmap，不在本变更中实现：

- 规则、技能、命令行工具的 root / project / service 三层作用域解析。
- 命令行工具清单默认叠加、同 `id` 下层整项覆盖和来源链输出。
- 工具型资产维护权限、审核和高风险控制。
- 更完整的技能引用命令行工具提醒策略。
- 更细的命令行工具检查状态和差异解释。

## Risks / Trade-offs

- [Risk] “工具型资产”被理解为统一管理模块。 → Mitigation：proposal、design、规则和 roadmap 都明确不新增统一资产命令，三类资产仍各自维护。
- [Risk] 用户期望 Buildr 自动安装命令行工具。 → Mitigation：README、规则、技能和检查输出都明确 Buildr 管团队资产，不接管个人机器。
- [Risk] scope 语义让 Agent 混淆产品随包源和用户工作区资产。 → Mitigation：规则和技能明确用户工作区维护自身 `AGENTS.md`、`rules/`、`skills/`、`commands/`；产品仓随包源只在开发 Buildr 产品时修改。
- [Risk] 不检查登录态导致 doctor 显示没有错误但具体命令仍不可用。 → Mitigation：doctor 只承诺检查清单、可执行命令和版本；认证失败由具体任务和对应技能处理。
- [Risk] 安装提示过少，用户不知道怎么补齐工具。 → Mitigation：清单提供最小安装提示或官方链接，Agent 根据当前机器和用户授权继续协作。
