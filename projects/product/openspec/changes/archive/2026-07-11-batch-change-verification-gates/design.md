## Context

Buildr 当前已经具备产品级总验证入口，并要求完整验证绑定最终候选 Git tree；`task-finish` 也会在 tree 未改变时复用验证证据。这解决了集成后的重复 E2E，却没有规定 apply 期间每完成一个任务应该运行多大范围的验证。Agent 容易把“必须验证”理解为“每一步都运行全部检查”，也可能把工具返回的运行中 session 当成超时并再次启动命令。

本 change 横跨 Product Project 开发契约、随包任务 Skills 和 package 验证断言。约束应由 Buildr 自有流程层提供，而不是修改 Component 管理的外部 `openspec-apply-change` Skill。

随包任务 Skills 承载适用于用户 workspace 的通用分层协议，不硬编码 Buildr 产品命令；每个 workspace 或 Project 通过自身规则、OpenSpec 或开发文档声明专项检查和完整验证入口。Product Project 开发契约只补充 Buildr 自举时使用的 package check、临时 workspace E2E 和产品总验证语义。

## Goals / Non-Goals

**Goals:**

- 在不降低最终候选覆盖的前提下，减少 change 开发期间的重复 package check 和 E2E。
- 让 Agent 能按单任务、任务组、最终候选三个层级选择验证范围。
- 让长时间运行命令使用同一进程持续等待，避免重复启动。
- 让任务清单和收尾流程能够表达并复用候选验证证据。
- 用随包静态契约检查防止流程文本在后续演进中回退。

**Non-Goals:**

- 不删除或缩减产品级总验证入口包含的检查。
- 不实现通用测试调度器、缓存服务或 CI 平台。
- 不规定所有技术栈的具体单元测试命令。
- 不修改外部 OpenSpec workflow Skills 或其上游生成内容。
- 不允许以“批量验证”为由跳过高风险变更所需的即时安全检查。

## Decisions

### 1. 使用三级验证门禁

实现阶段分为：单任务最小反馈、任务组受影响范围验证、最终候选完整验证。单任务检查只覆盖语法、类型或直接相关的小范围测试；任务组边界合并执行相互关联的专项检查；所有实现、自然语言资产和 review 修订完成后冻结候选，才运行当前项目定义的完整验证。对 Buildr 产品仓，这一入口是产品总验证并包含临时 workspace E2E。

选择该模型是因为它同时保留快速反馈和最终覆盖。替代方案“完全不做中间验证”会让简单错误积累到最后；“每个任务运行完整验证”正是当前耗时问题的来源。

### 2. 用风险和依赖边界划分任务组

任务组不是固定数量的 tasks，而是共享实现区域、验证入口或失败影响面的连续工作。例如一组 Skill 文本修改和对应静态断言应先整体完成，再运行一次专项检查。安全边界、不可逆迁移或用户明确要求的检查仍可立即执行。

选择语义边界而不是固定每 N 个任务验证，避免机械分组掩盖真实风险。

### 3. 最终验证前设置候选冻结点

进入完整验证前，Agent 必须确认计划中的实现、文档、生成资产同步和 review 修订均已完成。完整验证通过后，只有候选内容改变才需要重验；commit、相同 tree 的集成、push 和清理继续复用结果。

这将现有“最终候选 tree 证据”向前延伸到 apply 阶段，而不引入新的持久化数据模型。

### 4. 失败时缩小循环，稳定后只做一次最终完整验证

完整验证失败后，修复循环优先执行失败项和受修复影响的专项检查，不在每次编辑后重新运行整套 E2E。候选再次稳定时执行一次最终完整验证。若修复改变了其他风险面，Agent必须把相应专项检查加入受影响范围。

### 5. 运行中句柄代表同一验证进程

工具返回 session、cell、process id 或“仍在运行”状态时，Agent 必须通过 wait、poll 或 resume 继续该进程。暂时没有新输出不构成重启理由；只有进程明确结束且需要重试时才可启动新命令。

该约束直接处理重复 E2E 最隐蔽的来源，并保持与不同 Agent 工具接口兼容。

### 6. 由 Buildr-owned Skills 包裹外部 OpenSpec 流程

分层验证协议进入 `task-triage`、`task-worktree`、`task-finish` 以及 Product Project 开发规则；Component 管理的 `openspec-apply-change` 保持上游内容不变。`task-triage` 负责形成合理任务组，`task-worktree` 负责开发阶段执行边界，`task-finish` 负责最终证据复用。

替代方案是直接 patch 外部 Skill，但这会破坏 Component ownership 和上游升级能力。

### 7. package 验证只做契约防回退

产品 package check 或现有验证脚本增加静态断言，确认 Skills/项目规则包含三级门禁、运行中进程复用、失败后专项重验和最终候选一次完整验证。该检查验证流程资产是否交付，不尝试模拟 Agent 的完整推理过程。

## Risks / Trade-offs

- [任务组划分过大导致错误较晚暴露] → 要求单任务保留最小反馈检查，并允许高风险边界即时验证。
- [Agent 将“避免重复”误读为“跳过验证”] → 明确最终候选必须完成完整验证，且候选改变后仍需重验受影响范围。
- [静态文本断言对措辞敏感] → 检查稳定的协议要点和结构，不绑定完整段落原文。
- [不同测试入口覆盖关系不透明] → 由当前产品 release checklist 和任务上下文声明上层入口覆盖范围；无法证明覆盖时不得擅自去重。

## Migration Plan

1. 更新 canonical specs 和 Buildr-owned 任务流程源。
2. 更新 Product Project 开发规则，保持高层不变量简洁。
3. 扩展 package/产品验证断言并运行受影响范围检查。
4. 完成所有实现与 review 后冻结候选，只运行一次最终产品总验证。
5. 通过现有 update/sync 机制将新版 Skills 投射到自举 workspace。

回滚时恢复 Skills、项目规则和对应验证断言即可；本 change 不迁移用户数据，也不改变 CLI 数据格式。

## Open Questions

无。首版使用自然语言协议和现有验证入口，不引入额外验证缓存或机器可读 coverage graph。
