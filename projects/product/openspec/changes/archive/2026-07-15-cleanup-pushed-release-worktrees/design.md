## Context

`task-worktree` 当前把所有任务统一纳入“未上线、未归档或未明确收尾前保留”的策略。普通开发任务需要为联调和修复保留上下文，但发布 worktree 通常只是从既有发布基线制作、验证并推送新发布分支的临时环境；远端分支确认后，本地副本不再承担任务状态。

## Goals / Non-Goals

**Goals:**

- 明确发布 worktree 的完成条件和默认清理动作。
- 保留存在明确后续本地动作时的例外。
- 不改变普通开发任务的保守保留策略。

**Non-Goals:**

- 不自动删除远端发布分支。
- 不改变 Git push、force push 或分支删除授权边界。
- 不要求所有发布操作都必须创建 worktree。

## Decisions

1. 将发布 worktree 视为 `task-worktree` 生命周期中的特例，而不是新增独立 Skill。创建、保留和清理仍由同一 owner 管理，避免与 `git-ops` 重复。
2. 以四个条件定义默认清理终态：远端发布分支已推送、远端 ref 与候选提交一致、worktree 干净、没有明确的后续本地构建/部署/修复/验证动作。
3. 默认清理仅覆盖本地 worktree 和本地发布分支；远端分支继续受 Git 授权边界保护。
4. 若保留，Agent 必须说明具体原因和下一步，避免使用泛化的“开发任务默认保留”掩盖已经完成的发布目标。

## Risks / Trade-offs

- [部分团队在 push 后仍从本地 worktree 部署] → 将明确后续部署作为保留条件，不机械清理。
- [发布分支未真正同步到远端便误删本地候选] → 清理前要求核对远端 ref 与候选提交一致。
