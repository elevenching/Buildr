## Context

Buildr 目前由 `render`、`skills render`、`sync` 和 Component 安装卸载分别处理运行时文件。它们使用的检查和清理逻辑不同，因此可能留下旧文件、旧安装计划或部分写入结果。

## Goals / Non-Goals

**Goals:**

- 让相关命令使用同一套运行时同步逻辑。
- 写入前完成检查，写入后确认结果已经同步完成。
- 删除失去来源的 Buildr 受管文件，保留用户文件。
- 让重复执行得到相同结果。

**Non-Goals:**

- 不增加权限或门禁系统。
- 不增加新的资产类型。
- 不设计 Project 独立运行时目录。
- 不要求运行时文件像源资产一样支持完整回滚。

## Decisions

### 1. 先算清楚，再统一写入

运行时同步先读取源资产和现有运行时，列出要新增、更新、删除、保留和报错的文件。全部检查通过后才开始写入。

不继续在各命令中分别补清理逻辑，因为那会再次产生行为差异。

### 2. 所有相关命令共用同步入口

`render`、`skills render`、`sync` 和 Component 生命周期调用同一套底层逻辑。命令可以处理不同范围，但文件检查、写入、清理和结果确认保持一致。

### 3. 按命令范围处理 Project

单 Project render 只读取 workspace 和当前 Project。workspace 全量 render 才读取所有 Project。

多个 Project 引入完全相同的 Skill 时只写一份；只有不同内容要写入同一路径时才报错。

### 4. 只清理 Buildr 受管文件

Buildr 只删除带有可验证管理标记、且已经没有来源的运行时文件和安装计划。用户文件保持不变；如果用户文件占用目标路径，写入前报错。

### 5. `sync` 分两步执行

`sync` 先在现有事务中更新源资产，提交成功后再同步运行时。运行时失败时保留已提交的源资产，并给出重新执行的办法。

### 6. Core 要求简明表达

Core 增加一条通用规则：面向用户说明事情时，优先使用常用、直接、简练的语言；只有准确表达所必需时才使用专业术语，并在需要时解释。

## Risks / Trade-offs

- 跨 Project 检查可能发现以前被执行顺序掩盖的冲突 → 报告冲突来源，让用户统一 Skill 内容或名称。
- 意外 I/O 失败仍可能留下部分运行时文件 → 运行时是可重建结果，重新执行同一命令完成修复。
- 多个命令切换到底层共用逻辑可能改变旧的边缘行为 → 用现有测试和临时 workspace E2E 固定兼容行为。
