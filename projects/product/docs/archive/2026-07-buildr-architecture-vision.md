# Buildr 框架发展纲领与设计架构

## 一、定位

### 新一代软件工程三支柱

AI 时代的软件工程由三个正交维度构成：

| 维度 | 职责 | 典型代表 |
|------|------|----------|
| 模型能力 | 智能底座，决定推理质量 | GPT、Claude、Gemini 等 |
| Agent 能力 | 执行引擎，决定交互效率 | Qoder、Claude Code、Codex、Trae 等 |
| 项目管理能力 | 上下文基座，决定产出质量 | **Buildr** |

Buildr 选择做第三个维度：**为项目构建优质的 AI 项目管理框架，使任意 Agent 都能随时接手项目管理与执行。**

### 框架类比

Buildr 之于 AI 时代的项目，如同 Spring 之于 Java 开发，也如 Maven 之于 Java 项目协作契约：

- **相似于 Spring**：都是框架，定义结构、约束和最佳实践，与具体 IDE/工具无关
- **相似于 Maven**：团队一旦采用，就形成跨 IDE、跨工具的项目契约；新工具要么原生理解该契约，要么通过约定入口学会使用
- **超越传统框架**：Spring 主要服务代码开发，Maven 主要服务构建和依赖；Buildr 覆盖想法、调研、立项、产品、研发、测试、部署、运维的全生命周期上下文
- **原因**：AI 的自然语言理解能力打通了“非代码环节”和“代码环节”之间的壁垒，使统一项目上下文框架成为可能

### 可移植性

```
Spring 可跑在 IntelliJ、VS Code、Eclipse 上
Buildr 可跑在 Qoder、Claude Code、Codex、Trae 上
```

只需一个轻量 runtime adapter，即可在不同 Agent 产品间无缝迁移。哪个模型+Agent 更好用或更经济，做个适配器就能切换。

在产品使用层面，采用 Buildr 也意味着团队承认一个跨 Agent 的项目协作契约：

- Agent 原生支持 Buildr 时，可以直接读取 workspace 资产和规则。
- Agent 未原生支持 Buildr 时，用户先用 `buildr skill install <agent>` 安装 Buildr Skill，再通过自然语言表达“使用 Buildr 管理项目”；Skill 不可用时，Agent 读取 Buildr bootstrap guide 后调用 CLI 完成初始化、接入和诊断。
- `AGENTS.md`、OpenSpec、practices、skills 和 service metadata 是契约资产；`.claude/`、`.trae/`、`.cursor/`、`.codex/` 等只是对具体 Agent 的 runtime 投射。

## 二、核心设计原则

### 原则 1：渐进式披露（Progressive Disclosure）

**由模型架构决定的框架级约束，地位等同于 Spring 中的 IoC。**

大语言模型是自回归模型，只有一个无记忆的上下文窗口。上下文内容越与当前任务相关，模型输出质量越高。因此：

- AGENTS.md 只做总领和索引，不堆砌具体规则
- 具体规则、知识、模板拆分为独立的聚焦文件
- Agent 根据任务相关性逐层按需加载子文件
- 文件开头提供自描述摘要，便于 Agent 快速判断相关性
- 文件间引用显式化，构成框架自带的路由表

信息分层模型：

| 层级 | 内容 | 披露方式 |
|------|------|----------|
| 第 1 层 | "有什么"——索引、总领 | 每次任务必读 |
| 第 2 层 | "是什么"——某主题的具体规则或规范 | 按任务相关性选择性加载 |
| 第 3 层 | "怎么做"——示例、模板、详细步骤 | 真正执行时才加载 |

### 原则 2：单一事实来源（Single Source of Truth）

Buildr 的文件系统（Markdown + Git）是项目上下文的唯一权威源。所有 Agent 的知识产出最终都汇入 Buildr，所有 Agent 都从 Buildr 读取上下文。不存在分散的、不一致的多份数据。

### 原则 3：Agent 无关性

框架不依赖任何特定 Agent 产品的专属格式或 API。载体是 Markdown/YAML——所有大模型的通用语言。任何能读文件的 Agent 都能消费 Buildr 的内容。

### 原则 4：可审计演进

规则和记忆存在 Git 中，具备版本历史、diff、blame。每一次框架变更都可追溯：谁在什么时候、因为什么原因、改了什么规则。

## 三、架构模型

### 层级结构

```text
框架 → 组织（最高资产与权限边界） → 项目 → 代码服务
```

每层抽象出该级别的通用规则和实践。越上层越通用，越下层越具体。Agent 进入哪层目录，就自动获得该层及上层的上下文。

### 三层记忆架构

采用经典认知科学的三层记忆模型，沿三个正交维度切分项目知识：

| 记忆类型 | 维度 | 回答什么 | 说明 |
|----------|------|----------|------|
| **情景记忆** | 时间 | 发生了什么 | 记录变更、决策、事件的时间线，追溯"事"的演进过程 |
| **语义记忆** | 状态 | 当下是什么 | 事实记忆——业务事实与实现事实（代码仓库） |
| **程序记忆** | 能力 | 如何做得好 | 技能记忆——角色、原则、技能、模式、工具、实践 |

#### 情景记忆

针对"事"（业务变更）记录清楚时间与变更内容，方便追溯事的演进。

- 载体：`openspec/changes/`
- 内容：需求提案、设计决策、任务进展、归档记录
- 特征：有时间线、有因果、有上下文

#### 语义记忆

当下事实是什么。分为两个可选子层：

```
语义记忆
├── 业务层 — 系统是什么、业务 SHALL 做什么
└── 实现层 — 代码如何组织、技术架构是什么
```

- 业务层载体：`openspec/specs/` + `openspec/knowledge/`
- 实现层载体：Agent 知识提取产出（如 RepoWiki），通过适配层映射到 Buildr 结构
- 特征：描述当下状态，随系统演进而更新
- 说明：有些项目业务就是实现本身，此时两层合一

#### 程序记忆

如何更好完成这个事的能力。包括：

- 角色定义（"后端工程师"的身份和职责）
- 原则与规范（编码规范、设计原则、协作约定）
- 技能与模式（代码模板、工具类用法、构建命令、反例经验）
- 导航与披露（AGENTS.md 的渐进式索引本身也是程序记忆——它是让项目"做得更好"的方式）

载体：`AGENTS.md` + `rules/` + `package/` + `practices/` + `skills/` + templates

#### 记忆对应当前 Buildr 结构

| 三层记忆 | Buildr 载体 |
|----------|-------------|
| 情景记忆 | `openspec/changes/` |
| 语义记忆（业务层） | `openspec/specs/` + `openspec/knowledge/` |
| 语义记忆（实现层） | Agent 知识提取产出 → 映射写入 Buildr |
| 程序记忆 | `AGENTS.md` + `rules/` + `package/` + `practices/` + `skills/` + templates |

#### 扩展性机制

AI 时代的框架扩展性，不需要传统代码模式（注解、AOP、反射）。由于 Agent 具备智能理解能力，只需三种根本性机制即可保证框架的完整扩展性：

| 机制 | 本质 | 示例 |
|------|------|------|
| **Hook** | 在工作流特定时机插入自定义动作 | 推代码前跑验证、创建 change 时自动生成模板文件 |
| **Hard constraint（脚本）** | 用脚本/命令强制执行，结果可程序化验证 | `openspec validate --strict`、`mvnd compile`、Python 校验脚本 |
| **Soft constraint（规则）** | 写自然语言规则让 Agent 遵循，靠模型智能执行 | AGENTS.md 中的编码规范、设计原则、命名约定 |

三种机制覆盖了用户提高框架质量的所有杠杆：定义工作流时机、写可执行验证、写自然语言规则。

## 四、Buildr 与 Agent 的异构协作

### 核心关系

```
Buildr（项目框架）= 项目的大脑（存储的知识体系）
Code Agent（执行引擎）= 项目的手脚（可执行的能力引擎）
```

Buildr 定义"应该知道什么、遵循什么、具备什么能力"；Code Agent 提供"工具、运行时、执行力"来真正做事。两者是非侵入式的协作关系。

### Runtime render 架构

```text
┌─────────────────────────────────────────────┐
│           Buildr（资产源 / 单一事实来源）       │
│  AGENTS.md / practices/ / openspec/ / skills/│
└─────────────────────┬───────────────────────┘
                      │ resolve(scope) + render(target)
           ┌──────────┼──────────┐
           ↓          ↓          ↓
        Qoder      Claude     Codex
        runtime    runtime    runtime
```

- **资产源**：组织和项目的规则、OpenSpec、practices、skills 保存在 Buildr 目录中。
- **runtime 产物**：`.claude/`、`.cursor/`、`.qoder/`、`.codex/`、`CLAUDE.md` 等由 Buildr 根据当前入口渲染生成，不作为资产源。
- **生产侧映射**：Agent 专属能力（如 RepoWiki、知识卡片提取）通过 hook/映射规则，将产出写入 Buildr 的统一记忆结构。
- **不做双向同步**：runtime 可以重建；长期资产只以 Buildr 为准。

核心口径：Install to Buildr, render to Agent runtime。

### 适配层极简设计

每个 Agent 产品只需：
1. 一个**runtime adapter**——把 Buildr 解析出的规则、Skills 和桥接入口渲染到该 Agent 的原生 runtime 位置
2. 一组**生产侧映射规则**——定义“我的哪个产出对应 Buildr 的哪个记忆位置”

adapter 本身不保存长期资产，只负责把 Buildr 资产投影成当前 Agent 能加载的格式。

### 角色与 Agent 的关系

Buildr 定义的角色（`agents/`）是 **可选叠加层**，与 Code Agent 自带的 subagent 并存：

```
Code Agent（如 Qoder）
├── 自带 subagent（专家团）—— Agent 产品提供的工具型能力
│   ├── SearchAgent
│   ├── DebugAgent
│   └── CodeReviewAgent
│
└── 可选：加载 Buildr 定义的角色
    ├── 手动激活 → 用户在交互中指定"以后端角色工作"
    ├── 搬入 subagent → 把 Buildr 角色能力导入 Code Agent 的专家团
    └── 不用 → 直接用 Code Agent 自带能力，Buildr 角色静默
```

**当前规则**：只有用户显式指定角色时，才加载 `agents/` 下的角色文件。不指定时，Code Agent 以自身能力工作。

**演进方向**：程序记忆足够丰富时，角色可以被抽出为独立的专家 Agent，通过 multi-agent 编排完成端到端开发。

### 程序记忆的具象化光谱

程序记忆不是一种固定形态，而是从"文件"到"独立 Agent"的连续谱：

```
轻量 ────────────────────────────────────────→ 重量

规则文件         Skills         角色 system-prompt      独立 Agent
(被读取)       (被触发/加载)   (被加载为人格)          (被调度执行)

"编码用驼峰"    "飞书操作"     "后端工程师"           "Backend Agent"
```

从左到右，程序记忆越来越"活"——从被动的文本变成主动的执行体。

### 多 Agent 场景下的记忆分配

| 记忆类型 | 在多 Agent 协作中的角色 | 共享性 |
|----------|------------------------|--------|
| 情景记忆 | 所有 Agent 共享的任务上下文（"这次在做什么 change"） | 全局共享 |
| 语义记忆 | 所有 Agent 共享的事实基底（"系统当下是什么"） | 全局共享 |
| 程序记忆 | 按角色分配给对应 Agent | 部分共享 + 部分专属 |

程序记忆进一步拆分：

```
程序记忆
├── 通用层（所有 Agent 共享）—— Git 规则、提交规范、worktree 流程
├── 角色层（专属某类 Agent）—— BE 的编码规范、FE 的组件模式、QA 的测试策略
└── 技能层（可组合分配）—— 某些 skills 归 BE，某些归 FE，某些通用
```

### Skills 的两种消费方式

| 方式 | 机制 | 适用场景 |
|------|------|----------|
| **直接读取** | Code Agent 通过 AGENTS.md 渐进式披露，按任务去读相关的 `skills/`、`practices/`、templates，作为 soft constraint | 通用方式，任何 Agent 都能工作 |
| **runtime render** | Buildr 把 `skills/` 等资产渲染到 Code Agent 的原生 runtime 目录 | 利用 Agent 产品的原生技能系统获得更好的发现和加载体验 |

两者不互斥：
- 直接读取是**基础能力**——任何能读 Markdown 的 Agent 都能用
- render 是**增强能力**——利用特定 Agent 产品的原生机制（如自动触发、场景匹配）

### 非侵入式原则

Buildr 与 Agent 的协作始终保持非侵入性：

1. **最小依赖**：Code Agent 只要能读文件、遵循 AGENTS.md 指引，就能使用 Buildr 框架
2. **按需增强**：想要更好的体验，可以把角色搬入 subagent、把 Skills 渲染到 Agent 原生 runtime
3. **不做绑定**：Buildr 不要求 Code Agent 必须支持某种特定的 subagent 协议或 skill 格式

## 五、自进化模型

### 传统框架进化

```
社区实践 → 收集到组织 → 组织设计开发 → 发版 → 用户升级
```

### Buildr 进化

```
用户在项目中产生好实践
    → 告诉 Agent 优化框架文件 → 本地立即生效
                                     ↓
                          反哺到框架维护组织 → 评审发布 → 更多人受益
```

由于框架载体是 Markdown 文件（Agent 能直接读写），框架迭代周期可以从"版本发布"压缩到"一次对话"。

核心流程：
1. **日常使用中发现改进点**——更好的规则、更有效的实践、缺失的约定
2. **就地进化**——直接让 Agent 修改框架文件，立即在当前项目生效
3. **反哺框架**——将经过验证的改进提交到框架仓库，经维护组织评审后发布
4. **正向飞轮**——每个项目的实践都能丰富框架，框架反哺所有项目

## 六、文档寻址与存储演进

### 问题

当前 Buildr 的文档引用基于文件路径。路径在目录重构时会大面积失效，导致维护成本和断裂引用。

### 演进路径

```
当前：文件名引用 → 中期：文档 ID 索引 → 远期：结构化存储
```

| 阶段 | 寻址方式 | 载体 | 状态 |
|------|----------|------|------|
| 当前 | 文件名 + 相对路径提示 | Markdown 文件 | 已实施 |
| 中期 | 文档 ID（frontmatter 或索引表） | Markdown + 索引文件 | 规划中 |
| 远期 | 文档 ID + 结构化存储 | SQLite / 轻量数据库 | 构想 |

### 当前策略

1. **同仓库内**：使用文件名作为稳定标识（如 `task-triage.md`），不写完整路径
2. **跨仓库/跨层级**：用描述性语言（如“当前项目 AGENTS.md 指引的前端技能文档”）
3. **实践引用**：仅用文档名（如 `backend-java.md`），不写父目录

### 中期构想：文档 ID

每个文档在 YAML frontmatter 中声明唯一 `id`：

```yaml
---
id: task-triage
title: 任务分流判定规则
---
```

引用时使用 `id` 而非路径。索引表（或未来的应用）负责 `id → path` 的映射。

### 远期构想：结构化存储

Buildr 文档存入轻量数据库（如 SQLite），实现：

- 文档寻址不依赖文件系统路径
- 全文检索、关系查询、版本跟踪
- Agent 上下文装配可通过查询而非遍历目录
- 文件系统仍为导出/审计载体，数据库为运行时索引

## 七、发展路线

### 当前阶段：软件开发环节

以研发为核心，覆盖需求→设计→开发→测试→部署的完整链路。

### 未来扩展方向

由于 AI 的智能性打通了各环节壁垒，Buildr 可逐步扩展到：

```
想法 → 脑暴 → 调研 → 立项 → 产品 → 研发 → 测试 → 部署 → 运维
```

每个环节的工作产出，都可以被其他环节的 Agent 加载为上下文，降低环节间的信息传导损失。

### 终极形态

当框架成熟到一定程度，它不必局限于软件开发——任何需要"多环节协作、长期知识积累、多角色协同"的项目，都可以使用 Buildr 作为项目管理框架。

---

## 八、核心理念总结

1. **上下文即基础设施**——项目的上下文组织质量，决定了 AI 产出质量
2. **文件即协议**——Markdown + Git 是最通用、最可移植、最可审计的载体
3. **渐进式披露**——信息按需加载，不污染上下文窗口
4. **单向桥接**——Agent 能力产出流入 Buildr，所有 Agent 从 Buildr 消费
5. **智能自进化**——框架在使用中持续进化，迭代周期压缩到一次对话
6. **Agent 无关**——框架不绑定任何模型或 Agent 产品，支持零成本迁移
