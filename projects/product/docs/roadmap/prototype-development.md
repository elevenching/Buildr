# 原型开发能力设想

> Roadmap：尚未实现。Buildr 当前没有提供、注册或渲染原型开发 Skill、CLI 或受管工作流；本文仅保留未来产品方向。

未来的原型开发方向是把前端代码纳入项目上下文，由 Agent 根据业务需求先实现一版以静态数据和页面交互为主的代码原型。原型通过评审后，再交给设计和前端从代码层面继续专业化开发。

## 候选定位

原型开发可被定义为前端页面开发的一种探索形式，目标不是交付生产功能，而是验证产品设计、信息架构、页面状态和关键交互。

| 原型开发候选 | 正式前端开发 |
| --- | --- |
| 使用静态数据或模拟接口 | 连接真实 API 和后端契约 |
| 验证页面结构、交互和状态 | 实现完整功能、性能、兼容和工程质量 |
| 作为设计和前端接手的代码化交接物 | 作为可发布生产代码的一部分 |

## 候选适用条件

- 需求存在明显页面、流程或交互不确定性。
- 仅靠文档、截图或口述难以评审体验。
- 项目已有可运行的前端服务仓，且服务仓 `AGENTS.md` 说明了启动、构建和目录结构。
- 用户明确希望先做原型，或在提案通过前允许探索性原型。

该方向不适用于纯后端能力、纯数据修复或无 UI 变化的工程任务。

## 候选输入材料

未来实现可按任务相关性读取：

1. 当前项目 `AGENTS.md` 和前端服务仓 `AGENTS.md`。
2. 当前 change 的 `proposal.md`、相关 `specs/<capability>/spec.md`、`design.md` 和 `tasks.md`。
3. 现有页面、组件、路由、请求封装和样式体系。

## 候选流程

### 1. 确认原型范围

明确页面入口和目标用户、核心与分支流程、异常状态、关键字段、权限差异、提示口径和不做范围。

### 2. 构建静态交互

- 使用项目已有组件、样式、路由和状态管理方式。
- 使用静态数据、mock 数据或本地模拟接口。
- 覆盖正常、空态、加载、错误、禁用和权限差异等关键状态。
- 不为原型引入新的长期架构、组件库或并行技术栈。

### 3. 本地验证

- 按服务仓 `AGENTS.md` 启动前端项目。
- 在浏览器中走通核心路径和关键边界状态。
- 记录无法验证的部分和原因。

### 4. 评审和交接

未来流程需要说明原型覆盖的页面、流程和状态，静态数据与真实 API 的差异，需要设计补充的细节，以及正式开发时需要替换、删除或重构的临时代码。

## 候选验收标准

- 能演示本次需求的核心交互和关键状态。
- 与项目现有页面风格和组件体系基本一致。
- 静态数据足以支撑评审，不误导为真实接口结果。
- 明确标识原型代码边界，避免被误当作完整生产实现。

该方向进入实现前，需要独立 OpenSpec change 明确触发方式、资产边界、与现有前端服务仓和 task worktree 的关系、验证要求及原型代码退出策略。
