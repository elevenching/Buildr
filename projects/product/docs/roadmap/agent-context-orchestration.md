# 多 Agent 任务编排与上下文管理

> Roadmap：尚未实现。重要性：t0。紧急性：t1。本文只记录产品方向，不是当前能力或实施契约。

## 目标

Buildr 为 Agent 管理可检索、可流转的高质量上下文，使 Agent 能依据任务依赖并发执行，并减少上下文爆炸和重复探索。

核心模型：

```text
Specs（业务认知）
+ Codebase Memory（代码认知）
+ Task State（当前任务认知）
→ Agent 推理与执行
```

Buildr 负责上下文资产、索引、生命周期和任务间流转；Agent 负责按当前任务检索、选择和组装上下文。

## 模块与任务

### 1. 任务依赖与并发调度

- 从 Proposal、Design 和 Tasks 分析任务依赖，生成可审阅的执行 DAG。
- 只调度依赖已满足的节点，并对独立节点并发执行。
- 维护节点状态、失败传播、重试、取消、冲突和最终验证边界。
- 明确主 Agent 的全局目标维护、调度、结果整合和最终验证职责。

### 2. 动态执行能力

- Agent 不绑定前端、后端、架构或测试等固定岗位。
- 根据任务动态加载相关 Rules、Skills 和 Packages，形成当次执行能力。
- 定义能力包的发现、组合、冲突处理和适用范围。

### 3. 上下文资产与检索

- 以 Specs 保存业务目标、规则、设计和验收标准。
- 接入 Codebase Memory，为 Agent 提供由代码提炼的结构、职责、关系和历史知识；代码仍是真实实现来源。
- Manifest 仅保存仓库位置、服务登记、验证入口等少量稳定元数据，不承担完整业务或代码依赖图。
- 为 Agent 提供按任务检索和渐进加载入口，不由 Buildr 预先生成固定 Context Package。

### 4. 任务认知流转

- 定义 Task State，保存当前任务产生的关键决策、原因、排除方案、约束、发现、风险和验证结果。
- 支持下游节点继承单个或多个上游节点的 Task State，并保留来源与可追溯性。
- 研究多父节点合流时的选择、去重、冲突检测、合并和保真策略。
- 允许下游按需回溯上游产物与证据，避免只传摘要导致信息损失，也避免传递完整对话造成上下文爆炸。

### 5. 上下文质量与效果验证

- 定义上下文相关性、完整性、时效性、来源可信度和 token 成本指标。
- 评估重复检索率、任务返工率、并发收益和上下文压缩损失。
- 建立单 Agent 与 DAG 调度、直接检索与 Codebase Memory 辅助的对照验证。

## 首要探索顺序

1. 定义 Task DAG 与 Task State 的最小协议。
2. 验证多父节点上下文继承、证据回溯和冲突处理。
3. 接入一个 Codebase Memory 原型，验证 Specs + Codebase 的检索效果。
4. 在人工确认 DAG 的前提下实现分层并发调度。
5. 验证有效后，再推进自动拆解、自动调度和更广泛的 runtime 适配。
