# 后端工程师角色设计候选

> Roadmap：尚未实现。本文保留未来 BE 角色 Agent 的职责设想，不是当前 Rule、Skill、Agent runtime 资产或可直接执行的角色协议。

## 候选工作前置

未来实现需要决定如何让角色获得以下上下文，而不是由本文直接触发加载：

1. 当前组织、项目和服务仓，以及当前目录和上层相关 `AGENTS.md`。
2. 相关项目 `openspec/knowledge/`、`openspec/specs/` 和当前 `openspec/changes/<change-id>/`。
3. 用户任务意图和语义影响；只有改变业务语义时才维护对应 change 交付物。
4. 代码开发所需的任务 worktree。

技术栈、构建命令和包路径应以组织级、项目级和服务仓 `AGENTS.md` 为准。

## OpenSpec 协作设想

未来后端角色可在 change 中补充：

- 影响范围：模块、接口、数据库、消息、配置、定时任务、外部依赖。
- 设计决策：架构、跨服务、数据迁移或兼容策略。
- 规范变化：API、Schema、消息格式、安全策略或业务行为。
- 验证方式：构建、测试、联调、数据校验和回滚关注点。

## 实现原则候选

- 遵循项目既有分层、包结构和命名风格；不为未确认需求提前设计复杂抽象。
- 金额、数量等精度敏感字段使用项目语言和框架中适合的精确类型。
- 多表写入、资金、库存、状态流转等一致性敏感逻辑需要明确事务边界。
- 日志使用项目既有日志框架和正确级别，避免输出敏感信息。
- 安全改动关注鉴权、租户隔离、越权访问、输入校验和注入风险。

具体实践应来自当前组织或服务的 Rule 与 Skill，不能由通用角色草案覆盖。

## 验证与交付设想

未来角色应优先运行与改动范围匹配的项目命令，并在无法执行测试时说明原因和替代校验。若任务存在 OpenSpec change，应同步更新对应 `tasks.md` 勾选状态。
