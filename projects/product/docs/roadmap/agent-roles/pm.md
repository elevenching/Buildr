# 产品经理角色设计候选

> Roadmap：尚未实现。本文保留未来 PM 角色 Agent 的职责设想，不是当前 Rule、Skill、Agent runtime 资产或可直接执行的角色协议。

## 候选职责

未来 PM 角色可帮助把业务目标转成可评审、可实现、可验收的 OpenSpec change：

- 澄清业务背景、目标、范围和非目标。
- 描述用户角色、业务场景、业务规则和异常口径。
- 维护验收标准、风险、依赖和未决问题。
- 推动 `tasks.md` 随进展更新。

## 候选工作前置

未来实现需要决定如何让角色获得以下上下文，而不是由本文直接触发加载：

1. 当前目录及上层相关 `AGENTS.md`。
2. 当前项目 `openspec/config.yaml`。
3. 相关 `openspec/knowledge/` 和 `openspec/specs/`。
4. `openspec/changes/` 中已有的相关或冲突变更。

## OpenSpec 协作设想

change 统一在 `openspec/changes/<change-id>/` 下管理。未来 PM 角色可关注：

- `proposal.md` 的业务背景、目标、范围、验收标准和影响说明。
- `tasks.md` 的进展更新。
- 按需组织原型和会议记录，但相应资产边界仍需后续设计。

## 需求写法候选

需求侧重业务需求，不写技术实现。业务背景、目标、范围、验收标准和影响说明写入 `proposal.md`；能力约束写入 `specs/<capability>/spec.md`。数据库字段、SQL、接口路径、状态码、表结构、类名和方法名等实现细节应留给设计与实现阶段。

## 原型与可视化设想

复杂流程可用 Mermaid 绘制流程图、状态图或时序图。未来编码式交互原型方向见[原型开发能力设想](../prototype-development.md)，该方向当前同样尚未实现。

## 待设计的决策与澄清边界

未来角色协议需要明确在以下情况如何暂停并取得用户确认：

- 业务范围、优先级或验收口径存在冲突。
- 需要在多个方案之间做产品取舍。
- 是否进入实现阶段。
- 用户明确要求先评审需求或 proposal。
