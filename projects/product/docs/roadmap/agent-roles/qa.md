# 测试工程师角色设计候选

> Roadmap：尚未实现。本文保留未来 QA 角色 Agent 的职责设想，不是当前 Rule、Skill、Agent runtime 资产或可直接执行的角色协议。

## 候选工作前置

未来实现需要决定如何让角色获得以下上下文，而不是由本文直接触发加载：

1. 当前组织、项目、服务仓和目标环境。
2. 当前目录及上层相关 `AGENTS.md`。
3. 当前 change 的 `proposal.md`、`design.md`、`tasks.md` 和相关 spec。
4. 测试对象、版本、环境、数据来源、依赖服务和验收口径。

## OpenSpec 协作设想

未来 QA 角色可在 change 中补充：

- 测试范围：新增、修改、回归和不测范围。
- 测试类型：功能、接口、权限、数据、兼容、性能、安全、异常和回滚验证。
- 验收标准：哪些检查通过后可认为 change 完成。
- 缺陷和风险：阻塞问题、遗留风险、上线关注点。

## 测试设计候选

测试设计可从业务规则和系统行为出发，覆盖：

- 主流程、分支流程、异常流程和边界条件。
- 权限、角色、租户、数据隔离和越权风险。
- 新旧数据、历史兼容、迁移或回滚场景。
- API 请求/响应、错误口径、幂等性、重复提交和并发风险。
- 前端关键交互、加载、空态、错误态、禁用态和兼容性。
- 与资金、库存、订单、审批、消息等关键链路相关的回归范围。

## 执行、自动化与质量结论设想

未来角色协议可要求记录版本、环境、账号、数据和依赖服务状态；缺陷包含复现步骤、实际结果、期望结果、影响范围和严重程度；无法执行的测试项明确原因。自动化应优先覆盖稳定、高价值、重复执行的回归场景，而不是为一次性验证引入新框架。

质量结论应说明已测和未测范围、通过项、失败项、阻塞项、遗留风险及是否满足验收标准。若任务存在 OpenSpec change，应同步更新对应 `tasks.md` 勾选状态。
